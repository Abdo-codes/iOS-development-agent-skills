# Arabic And English Copy Guidelines

Use these rules when writing or reviewing app copy for Arabic and English in Apple apps.

## Product Copy

- English copy should sound like native English, not like a literal back-translation from Arabic
- Arabic copy should sound natural and product-ready, not machine-translated or overly formal unless the brand explicitly wants that tone
- Keep buttons, settings labels, and empty states short and scannable
- Keep terminology consistent across the app

## Common Quality Checks

- Avoid mixing Arabic strings with English abbreviations unless the product intentionally uses them
- Avoid transliterating technical words when there is a clearer product term in Arabic
- Keep the same action hierarchy across languages even if the wording changes
- If a phrase is awkward in one language, rewrite it instead of mirroring the source syntax

## Placeholders And Labels

- Placeholders should be helpful but short
- Labels should not repeat the same meaning as placeholder text unless that is the established pattern
- Empty states should explain what the user gets after taking the next step

## Dates, Numbers, Units

- Do not manually embed locale assumptions into copy when a formatter should provide the value
- Prefer localized composition:
  - localized number
  - localized unit
  - localized sentence shell

## App Store Copy

- Treat Arabic and English store copy as separate marketing surfaces
- Do not translate keywords mechanically
- Recheck screenshot captions and feature bullets for line length in Arabic
- Ensure store-facing tone matches the in-app tone

## Red Flags

- Word-for-word translation that feels stiff
- English strings shown with Arabic-only formatting
- Arabic strings shown in an English visual hierarchy
- Different feature meanings between Arabic and English versions of the same screen
